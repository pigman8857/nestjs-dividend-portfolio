## Performance Baseline

### Methodology

- **App:** local dev machine (Thailand) → NestJS (localhost:3000) → ADB Singapore
- **Database:** Oracle ADB Always Free tier (1 OCPU, 20 GB) — `ap-singapore-1`
- 5 runs per read endpoint, 3 runs per write endpoint
- Reported as raw `time_total` from curl (`-w "%{time_total}s"`)
- ADB TCP connect time confirmed at ~37ms (ICMP ping blocked by OCI firewall — normal)

### Network Floor

| Measurement | Value |
|-------------|-------|
| TCP connect to ADB (`ap-singapore-1`) | ~37ms |
| NestJS + Mongoose processing overhead | ~0–7ms |

The ~37ms TCP connect is the hard floor for every database operation from this location.

### Performance Analysis (from infra review)

**Reads and simple writes (~37–40ms) — expected and healthy.**
NestJS + Mongoose overhead is 0–7ms. Everything is bounded by the Singapore RTT. No further optimization is possible at the app layer from this location.

**ACID transaction cost (~182–223ms) — structural, not a bug.**
With `retryWrites=false` and `loadBalanced=true`, each transaction executes as sequential round-trips over TLS:
`begin → write users → write portfolios → write trades → commit` = 5–6 RTTs × ~37ms = 185–222ms.
This matches exactly what the connection parameters require and is not fixable at the application layer.

**What would actually move the needle:**

| Change | Expected impact |
|--------|----------------|
| Collocate NestJS in OCI `ap-singapore-1` (same region) | RTT drops to ~1–2ms → transactions drop to ~10–20ms |
| Batch Time Series inserts (multiple ticks per request) | Fewer round-trips per ingestion run |

**Verdict for current use (dev/portfolio app):** Performance is perfectly acceptable. Reads are responsive. Transaction latency is a known characteristic of cross-region TLS, not a defect.
