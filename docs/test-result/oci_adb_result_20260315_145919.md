# OCI ADB MongoDB Compatibility Test

**Date:** 2026-03-15
**Time:** 14:59:19 (UTC+7)
**Environment:** Oracle OCI Autonomous Database (ADB) — MongoDB-compatible API
**App:** MongoOracle (NestJS), `npm run start:dev`
**Base URL:** `http://localhost:3000`

---

## Background & Reason for Testing

This test was conducted to verify the full integration stack after a series of Terraform infrastructure bugs were discovered and fixed during the initial OCI ADB provisioning. The app could not connect to ADB at all before these fixes. The test confirms every layer — connectivity, authentication, schema access, and all application features — works correctly on the repaired infrastructure.

---

## Infra Issues Found and Fixed (Pre-Test)

Six bugs were identified and resolved in the Terraform configuration before this test was run.

### Issue 1 — Wrong ADB Hostname

**Problem:** `main.tf` constructed the MongoDB connection hostname as:
```
<adb_name>.adb.<region>.oraclecloud.com
```
OCI actually assigns a tenancy-prefixed hostname on a different domain:
```
<tenancy_prefix>-<adb_name>.adb.<region>.oraclecloudapps.com
```
The hardcoded pattern never resolved, causing all connection attempts to fail immediately.

**Fix:** Stopped constructing the hostname manually. Instead, read `connection_urls[0].mongo_db_url` directly from the ADB resource output and extracted the host via regex.

---

### Issue 2 — DB User Never Created (Silent Provisioner Failure)

**Problem:** A `null_resource` provisioner was used to create the MongoDB user via OCI CLI + `sqlplus`. Neither tool was installed on the runner. The script exited `0` (success) silently, so Terraform reported provisioning as complete — but no user was ever created.

**Fix:** Replaced the provisioner with `curl` POSTing SQL statements directly to the ORDS REST SQL endpoint:
```
POST /ords/admin/_/sql
```
This requires no additional tooling and gives a clear HTTP error response on failure.

---

### Issue 3 — ORDS Schema Not Enabled

**Problem:** After the user was created, the Oracle REST Data Services (ORDS) schema was never enabled for that user. The MongoDB wire protocol rejects connections for any user whose schema is not ORDS-enabled, with the error:
```
schema not enabled for ORDS
```

**Fix:** Added `ORDS.ENABLE_SCHEMA` to the provisioner SQL and to the manual fallback SQL script run after user creation.

---

### Issue 4 — Invalid `GRANT CREATE INDEX`

**Problem:** The provisioner attempted:
```sql
GRANT CREATE INDEX TO mongoapp;
```
`CREATE INDEX` is not a grantable Oracle privilege. This caused:
```
ORA-00990: missing or invalid privilege
```
and the entire provisioner script to fail.

**Fix:** Removed the grant entirely. In Oracle, a user can create indexes on any table they own — this is implicit with `CREATE TABLE` and does not require a separate grant.

---

### Issue 5 — Wrong MongoDB Database Name in URI

**Problem:** The Mongoose connection URI used:
```
mongodb://.../nestjs_dividend_portfolio
```
Oracle ADB's MongoDB-compatible API only allows a user to access their own schema. The database name in the URI must exactly match the Oracle username. Using any other name returns an authentication or access error.

**Fix:** Updated `main.tf` (output) and the NestJS `.env` file to set the database name to `mongoapp` (the Oracle DB username).

---

### Issue 6 — `compute_count` Drift on Always Free ADB

**Problem:** OCI automatically adjusts `compute_count` on Always Free ADB instances after provisioning. On subsequent `terraform apply` runs, Terraform detected a drift and attempted to update `compute_count` back to the declared value — OCI rejected this with a `403` error because Always Free resources cannot be reconfigured this way.

**Fix:** Added a lifecycle ignore rule in `main.tf`:
```hcl
lifecycle {
  ignore_changes = [compute_count]
}
```
